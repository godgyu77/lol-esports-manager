## 프롬프트 입력 형식

사용자는 아래 형식으로 업무를 지시한다. 모든 태그를 사용할 필요는 없으며, 상황에 맞는 태그만 사용한다.


| 태그       | 용도                    | 사용 시점      |
| -------- | --------------------- | ---------- |
| `[오류]`   | 에러 메시지, 버그 상황 (원문 포함) | 버그 수정 시    |
| `[요구사항]` | 원하는 결과 (결과 중심으로 작성)   | 항상         |
| `[현재동작]` | 지금 어떻게 동작하는지          | 동작 변경 시    |
| `[참고]`   | 참고할 기존 코드, 컴포넌트, 화면   | 기존 패턴 따를 때 |
| `[범위]`   | 수정 대상 파일/컴포넌트 한정      | 범위 명확할 때   |


### 효과적인 입력 원칙

- 에러 메시지는 해석하지 말고 **원문 그대로** 붙여넣는다.
- "거기", "그거" 대신 **파일명, 컴포넌트명, 함수명**을 명시한다.
- 비슷한 기존 구현이 있으면 **참고 대상을 지정**한다. (예: `OrderList처럼 해줘`)
- **현재 상태 + 기대 결과 + 위치** 세 가지가 포함되면 가장 정확한 응답이 가능하다.

## 작업 범위

- 요청 범위를 벗어난 수정 **금지**
- 관련 없는 리팩터링, 이름 변경, 파일 이동 **금지**
- 주변 코드 리팩토링, 주석 추가, 포맷팅 변경을 임의로 하지 않는다.
- 애매한 요구는 추정하지 않고 **확인 후 진행**

---

## 작업 방식

> 구체적 절차는 `pipeline.md`의 7단계 파이프라인을 따른다.

- **분석 우선**: 오류/요구사항을 받으면 먼저 원인을 분석하여 방안을 제시한다.
- **코드 수정은 허락 후**: 사용자가 "수정해", "고쳐", "적용해" 등 명시적으로 지시할 때까지 코드를 직접 수정하지 않는다.
- **코드 우선, 설명은 간결하게**: 수정 지시를 받으면 코드를 먼저 작성하고 부가 설명은 최소화한다.
- **최소 수정 원칙** 유지
- **고위험 변경**(공통 컴포넌트, 타입, API 인터페이스)은 영향 범위와 롤백 포인트를 사전 보고

---

## 기존 패턴 유지

- 프로젝트에 이미 존재하는 코드 스타일, 디렉토리 구조, 네이밍 규칙을 따른다.
- 새로운 패턴이나 라이브러리를 임의로 도입하지 않는다.

---

## 프로젝트 자동 인식

- 작업 시작 시 프로젝트의 **기술 스택**(package.json, pom.xml, build 설정 등)과 **디렉토리 구조**를 파악한다.
- 해당 프로젝트의 컨벤션에 맞춰 응답한다.
- 특정 기술 스택을 가정하지 않는다.

---

## 파일 생성 최소화

- 기존 파일 수정으로 해결 가능하면 새 파일을 만들지 않는다.
- 새 파일이 필요한 경우 기존 디렉토리 구조와 네이밍 규칙을 따른다.

---

## 에러 메시지 원문 포함

- 오류 분석 시 핵심 에러 메시지를 **원문 그대로** 인용하여 어디서 발생한 것인지 명확히 한다.

---
