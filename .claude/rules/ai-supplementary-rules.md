# AI 보완 규칙

> 💡 현재 팀 문서에 명시적으로 포함되어 있지 않으나, 최신 AI 개발 환경에서 필수적인 요소들.
> AI가 논리적으로 판단하여 추가한 보완 사항이다. (💡 표시)

---

## 💡 1. 컨텍스트 제한 극복법

> AI 모델의 컨텍스트 윈도우 한계를 실무에서 관리하는 전략

### 프롬프트 최적화

- 한 번에 하나의 작업만 요청한다. 여러 작업을 동시에 요청하면 품질이 저하된다.
- 긴 코드 파일은 **수정 대상 영역만** 발췌하여 제공한다.
- 이전 대화에서 결정된 사항은 **간결한 요약**으로 재전달한다.

### 대규모 코드베이스 전략

- 전체 파일 대신 **관련 함수/컴포넌트 단위**로 코드를 제공한다.
- 디렉토리 구조는 **트리 형태 요약**으로 제공하여 컨텍스트를 절약한다.
- 타입 정의와 구현부를 **분리하여 필요한 것만** 포함한다.
- 대화가 길어지면 새 세션을 시작하고 **핵심 컨텍스트만 재전달**한다.

### 메모리 활용

- 프로젝트 구조, 기술 스택, 반복 사용하는 패턴은 **메모리 파일**에 저장한다.
- 세션 간 지속되어야 하는 결정 사항은 `CLAUDE.md` 또는 프로젝트 메모리에 기록한다.

## 💡 2. 환각(Hallucination) 현상 방지

> AI가 존재하지 않는 API, 라이브러리, 메서드를 생성하는 문제 방지

### 방지 규칙

- **존재하지 않는 API/메서드를 생성하지 않는다**: 사용 전 반드시 실제 코드베이스에서 확인한다.
- **라이브러리 버전 확인**: `package.json` / `pom.xml`에 명시된 버전의 API만 사용한다.
- **불확실하면 명시**: "이 API가 해당 버전에서 지원되는지 확인이 필요합니다"라고 표기한다.
- **추측 코드 금지**: 확인할 수 없는 내부 API나 비공개 메서드는 사용하지 않는다.

### 검증 체크리스트

- [ ] import하는 모듈이 실제로 존재하는가?
- [ ] 사용하는 메서드가 해당 라이브러리 버전에서 지원되는가?
- [ ] 타입 정의가 실제 프로젝트의 타입과 일치하는가?
- [ ] 참조하는 파일 경로가 실제로 존재하는가?

## 💡 3. 코드 리팩토링 시 주의사항

> AI 기반 리팩토링에서 흔히 발생하는 문제와 방지책

### 리팩토링 원칙

- **동작 변경 없이 구조만 개선**한다 (기능 변경은 별도 작업)
- **한 번에 하나의 리팩토링 유형**만 적용 (이름 변경 + 구조 변경 동시 금지)
- **테스트가 없는 코드**는 리팩토링 전 최소한의 테스트를 먼저 작성한다
- 리팩토링 후 **모든 기존 테스트가 통과**하는지 반드시 확인한다

### 금지 사항

- 사용되지 않는 코드라도 **삭제 전 사용자 확인**
- **과도한 추상화** 금지: 재사용이 2회 미만인 로직의 공통화 금지
- 리팩토링과 기능 추가를 **같은 커밋에 혼합 금지**

## 💡 4. 보안 코딩 가이드라인

> AI 코드 생성 시 보안 취약점 방지

### 프론트엔드

- 사용자 입력을 **HTML에 직접 삽입 금지** (XSS 방지)
- `dangerouslySetInnerHTML` 사용 시 반드시 **sanitize** 처리
- API 키, 시크릿을 **클라이언트 코드에 노출 금지** (환경 변수 + 서버 프록시 사용)
- URL 파라미터, 쿼리스트링의 사용자 입력은 반드시 **검증/인코딩**

### 백엔드

- 파일 업로드 시 **확장자 + MIME 타입 + 크기** 검증
- ※ SQL 인젝션 방지, 에러 응답 보안, 인증/인가는 `language-java-springboot-rules.md` 참조

### 공통

- 민감 정보(비밀번호, 토큰, 키)는 **로그에 출력 금지**
- `.env`, `application-secret.yml` 등 시크릿 파일은 **`.gitignore`에 반드시 포함**
- 의존성 패키지의 **알려진 취약점** 정기 점검 (`npm audit` / OWASP Dependency-Check)

## 💡 5. AI 코드 생성 품질 관리

> AI가 생성한 코드의 품질을 보장하기 위한 추가 체크

### 생성 후 필수 확인

- **데드 코드**: 사용되지 않는 변수, import, 함수가 없는지 확인
- **하드코딩**: 환경/설정 값이 코드에 직접 박혀있지 않은지 확인
- **에러 핸들링**: 네트워크 요청, 파일 I/O 등 실패 가능한 연산에 에러 처리가 있는지 확인
- **엣지 케이스**: null, undefined, 빈 배열, 빈 문자열 등 경계값 처리 확인
- **메모리 누수**: 이벤트 리스너, 타이머, 구독 등이 정리(cleanup)되는지 확인

### 코드 리뷰 자동 트리거

- 10줄 이상의 코드 변경 시 → Code Reviewer 에이전트 활성화 권장
- 공통 컴포넌트/유틸 수정 시 → 전체 사용처 영향도 분석 필수
- API 인터페이스 변경 시 → 프론트/백 양측 호환성 확인

## 💡 6. Git 워크플로우 가이드

> AI와 협업할 때의 Git 브랜치/커밋 전략

### 브랜치 전략

- AI 작업은 **feature 브랜치**에서 수행 (main/develop 직접 수정 금지)
- 브랜치명: `feature/{작업유형}/{간단설명}` (예: `feature/bugfix/prompt-save-error`)
- AI 작업 완료 후 반드시 **사람이 리뷰** 후 merge

### 커밋 단위

- **하나의 논리적 변경 = 하나의 커밋**
- 리팩토링과 기능 변경은 **별도 커밋**으로 분리
- AI가 생성한 코드임을 커밋 메시지에 명시 (예: `Co-Authored-By: AI`)
- ※ 커밋 메시지 형식은 `language-react-typescript-rules.md` 커밋 규칙 참조

### PR 작성

- AI가 작성한 코드 변경 사항을 **명확히 설명**
- 영향 받는 파일과 테스트 결과를 포함
- 리뷰어가 **AI 생성 코드를 인지**할 수 있도록 표기
