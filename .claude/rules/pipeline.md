# Team AI Pipeline

> 신입/비숙련 사용자도 요청만으로 안정적으로 수정하고 최종 검토 품질을 확보하는 것을 목표로 한다.
> 출처: lhg pipeline.md + khg 수정전후 영향도 규칙 통합

---

## 7단계 순차 파이프라인

### 1. 요청 해석

- 목표/범위/비범위를 3줄 이내로 명시한다.
- 불명확한 조건이 있으면 바로 확인한다.
- **Action**: `[목표] / [범위] / [비범위]` 형식으로 정리하여 사용자에게 확인

### 2. 영향 분석

- 수정 후보 파일과 연관 파일을 식별한다.
- 구조/라우팅/API/상태 영향 여부를 표시한다.
- 해당 함수/컴포넌트/타입을 import하고 있는 파일 확인
- props, 파라미터, 리턴 타입 변경 시 호출부 확인
- 공통 컴포넌트/유틸/훅 수정 시 전체 사용처 확인
- **Action**: Grep/Glob으로 import 추적 → 영향 파일 목록 작성

### 3. 변경 계획

- 최소 변경 단위를 정의한다.
- 파일별 변경 의도를 간략히 정리하고 **바로 구현으로 진행**한다.
- 대규모 변경(5개 파일 이상)은 계획을 먼저 공유한 후 진행한다.
- **Action**: 파일별 `[파일명]: [변경 의도]` 목록 작성 → 즉시 구현 (소규모) 또는 확인 후 진행 (대규모)

### 4. 구현

- 공통 규칙(코어 규칙 + 도구 어댑터)을 따른다.
- 무관한 정리/리네이밍을 하지 않는다.
- 범위 외 수정이 필요하면 즉시 사용자 확인 후 진행한다.

### 5. 검증

- 타입체크, 린트, 테스트(가능한 범위)를 실행한다.
- 실패 시 원인/영향/대응을 정리한다.
- 변경된 파일을 import하는 곳에서 타입 에러가 발생하지 않는지 확인
- 빌드가 통과하는지 확인하고, 실패 시 원인과 수정 방안 제시
- **Action**: React → `tsc --noEmit` + `npm run lint`, Java → `mvn compile` 실행

### 6. 최종 리뷰

- 코드 품질 4원칙 체크
- 보안 규칙 체크 (→ `security-guidelines.md`)
- 성능 우선순위 체크
- 회귀 위험 체크

### 7. 결과 보고

- 변경 파일 목록
- 핵심 변경점
- 검증 결과 및 미실행 항목
- 후속 제안(필요 시)

## 품질 게이트

| 게이트 | 통과 조건 |
|--------|-----------|
| Gate 1 | 요청 범위 위반 없음 |
| Gate 2 | 구조/상태/API 규칙 위반 없음 |
| Gate 3 | 보안 규칙 위반 없음 |
| Gate 4 | 검증 결과 보고 누락 없음 |

검증/리뷰 없는 완료 보고를 **금지**한다.
