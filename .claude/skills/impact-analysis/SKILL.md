# Impact Analysis Skill

> 코드 변경 전 영향 범위를 체계적으로 분석하는 스킬.

---

## 언제 사용하나

- 공통 컴포넌트, 유틸, 훅을 수정하기 전
- 타입/인터페이스를 변경하기 전
- API(DB 쿼리, 엔진 함수) 시그니처를 변경하기 전
- 대규모 리팩토링 전 영향 범위를 파악할 때

---

## 분석 프로세스

### 1. 대상 식별

사용자가 지정한 파일/함수/타입을 확인한다.

### 2. Import 추적

대상을 import하고 있는 모든 파일을 찾는다.

```
Grep: import.*{대상} 또는 from.*{파일경로}
```

### 3. 사용처 분석

각 import 파일에서 실제 사용 방식을 분석한다:
- **함수 호출**: 파라미터, 리턴값 사용 방식
- **타입 참조**: extends, implements, 속성 접근
- **컴포넌트 사용**: props 전달, 이벤트 핸들러
- **re-export**: 다른 모듈에서 다시 내보내는 경우 (연쇄 영향)

### 4. 위험도 분류

| 등급 | 기준 | 조치 |
|------|------|------|
| **Critical** | 5개 이상 파일에서 사용, 또는 타입 시그니처 변경 | 신중한 수정 + 전수 검증 필요 |
| **High** | 3~4개 파일에서 사용, 또는 공통 컴포넌트 | 호출부 업데이트 필요 |
| **Medium** | 1~2개 파일에서 사용 | 일반적 수정 |
| **Low** | 사용처 없음 (dead code 가능성) | 삭제 검토 |

### 5. 연쇄 영향 확인

변경 대상이 다른 공통 모듈에서 사용되는 경우, 2차 영향까지 추적한다.

```
대상 파일 → 1차 사용처 → 2차 사용처 (re-export, 확장 등)
```

## 프로젝트 특화 체크포인트

### 엔진 (src/engine/) 수정 시
- 시즌 엔진(`src/engine/season/`)에서 호출하는지 확인
- 매치 시뮬레이터와의 연동 확인
- 밸런스 테스트 영향 여부

### 타입 (src/types/) 수정 시
- `src/types/index.ts` re-export 확인
- DB 쿼리(`src/db/`)에서 해당 타입 사용 확인
- Store(`src/stores/`)에서 캐시하는지 확인

### 컴포넌트 (src/components/, src/features/) 수정 시
- 라우터 등록 확인
- 부모 컴포넌트에서의 props 전달 확인
- Store 연동 확인

### DB 쿼리 (src/db/) 수정 시
- 엔진에서 호출하는지 확인
- 마이그레이션과 스키마 일치 여부
