# Play QA Checklist And Local QA Scope

작성일: 2026-04-06  
대상 프로젝트: `lol-esports-manager`

## 1. 목적

이 문서는 직접 플레이 QA를 전부 수행하기 어려운 상황을 전제로, 아래 두 가지를 함께 정리한다.

1. 사람이 실제로 플레이하면서 확인해야 하는 체크리스트
2. Codex가 로컬에서 대신 수행할 수 있는 자동 QA / 스모크 QA 범위

핵심 원칙은 다음과 같다.

- 자동 QA는 안정성, 회귀, 빌드 가능 여부를 빠르게 확인한다.
- 수동 QA는 재미, 피로도, 의사결정 무게, UX 명확성을 확인한다.
- 이 프로젝트는 매니저 시뮬레이션이므로 “기능이 동작하는가”보다 “선택이 의미 있게 느껴지는가”가 더 중요하다.

- 해석: 현재 드러난 것은 기능 실패라기보다 “로컬라이징 이후 테스트 기대값 정리 필요”에 가깝다

## 3. Codex가 할 수 있는 QA

Codex가 바로 수행 가능한 QA:

- 타입체크
- 빌드 검증
- 단위 테스트 / 릴리스 테스트 실행
- 특정 화면 기준 텍스트 회귀 확인
- 저장/로드/시즌 종료 같은 코드 경로 중심 점검
- 특정 기능의 테스트 추가 및 실패 원인 분석
- 앱 실행 스모크 확인
  - 실행 자체, 콘솔 오류, 초기 화면 진입 가능 여부 중심

Codex가 제한적으로만 가능한 QA:

- 실제 플레이 재미 평가
- 체감 난이도 평가
- 정보 과부하/피로도 평가
- “왜 이 선택을 해야 하는지”가 직관적인지 판단
- 장시간 수동 플레이 리듬 평가

즉, Codex는 “망가졌는지”는 잘 잡을 수 있지만, “정말 재미있고 자연스러운지”는 체크리스트 기반의 수동 플레이가 필요하다.

## 4. 추천 QA 방식

가장 효율적인 방식은 아래 순서다.

1. Codex가 자동 QA를 먼저 수행
2. 사람이 핵심 시나리오만 짧게 플레이
3. 이상 징후가 보인 구간만 다시 Codex가 원인 분석 및 수정

이 방식이면 전 화면을 다 수동으로 돌지 않아도 된다.

## 5. 플레이 QA 우선순위

### A. 진입 경험

목표:
- 처음 시작했을 때 무엇을 해야 하는지 바로 이해되는지 확인

체크:
- 메인 메뉴에서 `새 커리어 시작`과 `계속 진행`의 의미가 즉시 보이는가
- 감독/선수 모드 차이가 텍스트만 읽어도 이해되는가
- 첫 5분 안에 “이 게임의 핵심 재미”가 전달되는가
- 저장된 세이브가 있을 때 복귀 동선이 자연스러운가

실패 신호:
- 어디를 눌러야 할지 잠깐 멈추게 됨
- 문구는 많지만 시작 행동이 선명하지 않음

### B. DayView 코어 루프

목표:
- “오늘 제일 먼저 해야 할 일”이 명확한지 확인

체크:
- 세팅 미완료, 재정 압박, 후폭풍, 경기 준비 중 무엇이 최우선인지 즉시 보이는가
- 우선 액션 버튼이 실제로 기대한 화면으로 연결되는가
- 오늘의 리스크와 오늘의 가치가 같은 화면 안에서 이해되는가
- 하루 진행 전 “왜 지금 넘기면 안 되는지”가 보이는가

실패 신호:
- 카드가 많지만 우선순위를 읽기 어렵다
- 다음 행동이 명확하지 않아 화면을 왕복하게 된다

### C. 훈련/전술/준비

목표:
- 조정의 이유와 효과가 분명한지 확인

체크:
- 훈련 화면에서 이번 주 계획이 다음 경기와 어떻게 연결되는지 보이는가
- 전술 화면에서 바꾼 값이 “무슨 대가를 치르고 어떤 보상을 노리는지” 이해되는가
- 경기 전 준비 상태가 DayView, TrainingView, TacticsView 사이에서 일관되게 읽히는가

실패 신호:
- 변경은 가능하지만 왜 바꿔야 하는지 설득력이 약하다
- 여러 화면이 서로 같은 정보를 다르게 말한다

### D. 재정/계약/이적 압박

목표:
- 운영 게임다운 긴장감이 실제로 느껴지는지 확인

체크:
- 재정 활주로, 실패 협상 비용, 보드 반응이 실제로 압박처럼 느껴지는가
- 예산 압박이 DayView와 FinanceView에서 동시에 체감되는가
- 무리한 협상 이후 후폭풍이 화면에 남는가
- 돈 문제를 무시하고 진행했을 때 “아프다”는 느낌이 있는가

실패 신호:
- 숫자는 보이는데 심리적 압박은 약하다
- 잘못된 선택을 해도 큰 대가가 없어 운영 감각이 흐려진다

### E. 경기 경험

목표:
- 라이브 매치가 보기 좋고 읽기 쉬운 상태인지 확인

체크:
- 중계 HUD, 미니맵, 팀 패널, 채팅, 하이라이트 중 시선이 어디로 가야 하는지 자연스러운가
- 속도 조절, 전술 변경, 선수 지시가 실제 경기 흐름을 깨지 않고 읽히는가
- 경기 중 정보량이 많아도 “지금 무슨 일이 일어나는지” 놓치지 않는가
- 경기 후 요약이 충분히 납득 가능한가

실패 신호:
- 화면은 화려하지만 무엇을 봐야 할지 모른다
- 전술/지시 UI가 경기 감상 흐름을 끊는다

### F. 시즌 서사 / 기억성

목표:
- “기사 수”보다 “내 이야기”가 남는지 확인

체크:
- 최근 커리어 아크가 DayView나 커리어 화면에서 실제로 기억에 남는가
- 뉴스와 관계 변화가 반복 플레이 동기로 이어지는가
- 시즌을 넘긴 뒤 “다음 시즌을 보고 싶다”는 감정이 생기는가

실패 신호:
- 뉴스는 많지만 기억에 남는 인물/갈등/라이벌 구도가 없다

### G. 저장/로드/장기 진행

목표:
- 유료 상품 기준의 신뢰 확보

체크:
- 저장 후 즉시 복귀가 문제없는가
- 경기 직전, 경기 직후, 시즌 종료 직후 저장/로드가 안전한가
- 오프시즌 진입과 다음 시즌 시작이 자연스러운가
- 며칠 자동 진행 후에도 루프가 깨지지 않는가

실패 신호:
- 특정 타이밍 저장/로드가 불안하다
- 시즌 경계에서 데이터가 누락되거나 화면 흐름이 어색하다

## 6. 최소 수동 QA 시나리오

시간이 부족할 때는 아래 6개만 돌려도 좋다.

### 시나리오 1. 새 감독 커리어 생성 후 첫 3일 진행

확인 포인트:
- 진입 설명
- DayView 우선순위
- 기본 루프 이해도

### 시나리오 2. 훈련/전술 수정 후 스크림 또는 경기 직전까지 진행

확인 포인트:
- 변경 이유 체감
- 준비 상태 피드백

### 시나리오 3. 재정 압박 상태에서 하루 진행

확인 포인트:
- DayView 경고
- FinanceView 일관성
- 보드 반응

### 시나리오 4. 협상 실패 후 후폭풍 확인

확인 포인트:
- 비용 반영
- 압박도 상승
- 후속 메시지 노출

### 시나리오 5. 한 경기 플레이 후 경기 후 브리핑 확인

확인 포인트:
- HUD 가독성
- 경기 중 의사결정 UI
- 경기 후 요약 납득성

### 시나리오 6. 저장 후 종료, 재실행 후 복귀

확인 포인트:
- 세이브 안정성
- 복귀 동선

## 7. 자동 QA 체크리스트

릴리스 전 Codex가 수행할 기본 자동 QA:

1. `npm run typecheck`
2. `npm run build`
3. 핵심 테스트
   - `src/features/main/MainMenu.test.tsx`
   - `src/features/manager/pages/DayView.test.tsx`
   - `src/features/match/LiveMatchView.test.tsx`
4. 릴리스 테스트 묶음
   - `npm run test:ea`
5. 실패 테스트가 있으면
   - 기능 실패인지
   - 문자열 기대값 불일치인지
   - 테스트 fixture가 오래된 것인지
   구분해서 기록

## 8. 권장 판정 기준

### 출시 가능

- 빌드 통과
- 타입체크 통과
- 핵심 테스트 통과
- 릴리스 테스트에서 기능 실패 없음
- 수동 시나리오 6개 중 치명적 막힘 없음

### 수정 후 재검토

- 화면은 열리지만 흐름이 반복적으로 헷갈림
- 재정/전술/훈련의 무게감이 약함
- 경기 화면 정보 위계가 불명확함

### 출시 보류

- 저장/로드 불안정
- 시즌 전환 실패
- 경기 종료 후 진행 막힘
- 예산/보드/준비 시스템이 상호 충돌

## 9. 지금 가장 추천하는 다음 행동

가장 효율적인 다음 순서는 아래와 같다.

1. Codex가 `test:ea` 실패 항목 정리 또는 수정
2. 사용자는 최소 수동 시나리오 6개만 플레이
3. 플레이 중 어색했던 장면만 메모
4. Codex가 그 메모를 기반으로 다시 개선

## 10. 결론

직접 전체 플레이 QA를 다 하지 않아도 된다.

이 프로젝트는 아래 방식이 가장 현실적이다.

- Codex가 자동 QA와 코드 회귀 검증 담당
- 사용자는 핵심 6개 시나리오만 짧게 플레이
- 체감 이슈는 다시 Codex가 수정

즉, “전체 QA를 사람이 다 한다”가 아니라 “사람은 재미와 감각만 확인하고, 나머지는 Codex가 받쳐주는 구조”로 가는 것이 가장 효율적이다.
